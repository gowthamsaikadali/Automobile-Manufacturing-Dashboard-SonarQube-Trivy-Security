# Automobile-Manufacturing-Dashboard-SonarQube-Trivy-Security

Two-tier automobile manufacturing dashboard (Flask + RDS MySQL) on AWS EKS,
hardened with IAM least-privilege, Secrets Manager, ACM/TLS, WAF, Kubernetes
Network Policies + RBAC, and a CI/CD pipeline gated by SonarQube and Trivy.

This bundle gives you:
1. The Flask app (matches your dashboard screenshot) + MySQL schema
2. Docker image (hardened, non-root, Trivy-friendly)
3. Terraform for ECR, RDS, IAM (least privilege), Secrets Manager, ACM, WAF
4. Helm chart for EKS with RBAC, NetworkPolicies, ExternalSecrets
5. GitHub Actions pipeline with SonarQube + Trivy gates, OIDC deploy

**No AWS account ID, bucket name, or role ARN is hardcoded anywhere in
this repo.** Everything account-specific is either generated by
Terraform, or supplied at run time via GitHub Actions repo *Variables*
(Settings → Secrets and variables → Actions → Variables tab) or `--set`
flags you pass yourself. That keeps the repo safely shareable/public.

Everything below is spoon-fed, in order. Do not skip steps — later steps
depend on outputs from earlier ones.

## Step 0. Bootstrap Terraform remote state (one-time, before anything else)

`terraform/bootstrap` creates the S3 bucket + DynamoDB table that
`terraform/infra` needs as its backend. It uses a random suffix for
uniqueness instead of your account ID, and runs with local state only
(chicken-and-egg — nothing else exists yet for it to store state in):

```powershell
cd terraform/bootstrap
terraform init
terraform apply
terraform output state_bucket_name
terraform output lock_table_name
cd ../..
```

**Write down both output values** — you'll pass them into every future
`terraform init` for `terraform/infra`, and into GitHub Actions repo
variables later. They look like `autoforge-tfstate-a1b2c3d4` and
`autoforge-tf-locks-a1b2c3d4`.

## Step 1. Prerequisites (run once, on your Windows/PowerShell machine)

```powershell

```

You'll need: an AWS account with admin access to bootstrap the roles
below, and GitHub OIDC federation set up once per AWS account (Terraform
creates the *role* in Step 3; the OIDC *provider* itself — trusting
`token.actions.githubusercontent.com` — needs to exist first; see
"One-time: GitHub OIDC provider" below).

**You do not need an existing VPC, subnets, or EKS cluster** — Terraform
builds all of that from scratch in Step 3 (`network.tf` + `eks.tf`).

### One-time: GitHub OIDC provider (skip if your AWS account already has one)
```powershell
aws iam create-open-id-connect-provider `
  --url https://token.actions.githubusercontent.com `
  --client-id-list sts.amazonaws.com `
  --thumbprint-list 6938fd4d98bab03faadb97b34396831e3780aea1
```
If this already exists you'll get an `EntityAlreadyExists` error — that's fine, it means you're set.

## Step 3. Terraform — provision VPC, EKS, ECR, RDS, IAM, Secrets Manager, ACM, WAF

Everything in this step is built from nothing — no existing VPC, subnets,
or cluster required.

**Cost note**: this creates real billable resources — the EKS control
plane alone is ~$0.10/hr (~$73/mo) regardless of usage, plus 2×
`t3.small` nodes and a `db.t3.micro` RDS instance. Run `terraform destroy`
when you're done experimenting to avoid ongoing charges.

```powershell
cd terraform/infra
```

Set these `-var` flags (or edit `variables.tf` defaults):
- `tf_state_bucket_name`, `tf_lock_table_name` — the two values from Step 0
- `domain_name` — **leave this blank for now** (the default). You don't
  have a domain yet, so `acm.tf` skips creating a certificate entirely
  (it's conditional on `domain_name != ""`). The app is reachable over
  plain HTTP via the ALB's own auto-generated DNS name instead.

```powershell
terraform init `
  -backend-config="bucket=<state_bucket_name from Step 0>" `
  -backend-config="dynamodb_table=<lock_table_name from Step 0>"

terraform plan -out=tfplan `
  -var="tf_state_bucket_name=<state_bucket_name>" `
  -var="tf_lock_table_name=<lock_table_name>"

terraform apply tfplan
```

Grab a coffee — an EKS cluster + node group takes **10–15 minutes** to
come up. This creates, among other things:
- `aws_vpc.main` + 2 public subnets (EKS nodes, no NAT Gateway — saves
  ~$32/mo) + 2 private subnets (RDS only, no internet route needed)
- `aws_eks_cluster.main` — the EKS control plane (`autoforge-eks` by
  default; change `eks_cluster_name` in `variables.tf` if you want a
  different name)
- `aws_eks_node_group.main` — 2× `t3.small` worker nodes (adjust via
  `node_instance_type` / `node_desired_size`)
- `aws_iam_openid_connect_provider.eks` — enables IRSA (IAM Roles for
  Service Accounts), wired automatically into the app pod role
- `aws_ecr_repository.app` — a fresh, scan-on-push ECR repo
- `aws_secretsmanager_secret.rds_credentials` — the ONLY place DB creds live
- `aws_db_instance.mysql` — private-subnet-only, encrypted, TLS-required
- `aws_iam_role.eks_node_role` / `cicd_role` / `app_pod_role` — three
  **separate** least-privilege roles (satisfies "separate roles for EC2,
  EKS nodes, CI/CD")
- `aws_acm_certificate.autoforge` — **skipped** while `domain_name` is blank
- `aws_wafv2_web_acl.autoforge` — SQLi/XSS managed rules + rate limiting (works with or without a domain)

Point `kubectl`/`helm` at the new cluster:
```powershell
aws eks update-kubeconfig --name (terraform output -raw eks_cluster_name) --region ap-south-1
kubectl get nodes   # should show 2 nodes, Ready, after a couple minutes
```

Grab the rest of the outputs — you'll need these in Step 5 and Step 6:
```powershell
terraform output ecr_repository_url
terraform output cicd_role_arn
terraform output app_pod_role_arn
terraform output waf_web_acl_arn
```

### Getting a domain later (optional, whenever you're ready)
- Cheapest: register via Route 53 or Namecheap (~$1–15/yr for `.xyz`/`.online`)
- Free: a subdomain from [duckdns.org](https://www.duckdns.org) — supports the DNS TXT record ACM needs
- Set `domain_name` in your `-var` flags, uncomment the Route53 validation block in `acm.tf` (or validate manually via the CNAME ACM gives you), then `terraform apply` again.

## Step 4. Install External Secrets Operator on the cluster (one-time)

This is what fixes the empty `DB_HOST`/`DB_PASSWORD` bug class entirely.
Credentials flow: **Secrets Manager → ESO → native K8s Secret → pod env**
— never through a values file.

```powershell
helm repo add external-secrets https://charts.external-secrets.io
helm repo update
helm install external-secrets external-secrets/external-secrets `
  -n external-secrets --create-namespace
```

Also install the AWS Load Balancer Controller if not already present. It
needs its own IAM role (IRSA) — `terraform/infra/alb-controller-iam.tf`
creates this for you; grab the ARN and annotate the ServiceAccount with it
at install time (this is the step that's easy to miss and causes the
controller to crash-loop with "unable to introspect region from
EC2Metadata" — it has no credentials otherwise):
```powershell
cd terraform/infra
terraform output -raw alb_controller_role_arn
cd ../..

helm repo add eks https://aws.github.io/eks-charts
helm repo update
helm install aws-load-balancer-controller eks/aws-load-balancer-controller `
  -n kube-system `
  --set clusterName=autoforge-eks `
  --set region=ap-south-1 `
  --set vpcId=(terraform -chdir=terraform/infra output -raw vpc_id) `
  --set serviceAccount.create=true `
  --set serviceAccount.name=aws-load-balancer-controller `
  --set serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=(terraform -chdir=terraform/infra output -raw alb_controller_role_arn)
```

Verify it comes up healthy before moving on (give it ~30–60s):
```powershell
kubectl get pods -n kube-system | Select-String aws-load-balancer
kubectl logs -n kube-system -l app.kubernetes.io/name=aws-load-balancer-controller --tail=20
```
You want `1/1 Running`, not `CrashLoopBackOff`.

## Step 5. Deploy with Helm (account-specific values passed via --set, nothing committed)

```powershell
helm upgrade --install autoforge ./helm/autoforge `
  --namespace autoforge --create-namespace `
  -f ./helm/autoforge/values-dev.yaml `
  --set image.repository=<terraform output ecr_repository_url> `
  --set serviceAccount.roleArn=<terraform output app_pod_role_arn> `
  --set ingress.wafAclArn=<terraform output waf_web_acl_arn>
```

If you'd rather not type these every time, copy `values.yaml` to a local
`values-local.yaml` (already in `.gitignore` from Step -1), fill in the
three fields there, and use `-f values-local.yaml` instead of the `--set`
flags. It will never be pushed to GitHub.

## Step 6. GitHub Actions setup (repo Settings → Secrets and variables → Actions)

**Variables tab** (not secrets — these aren't sensitive, just account-specific):
- `ECR_REPOSITORY` — from `terraform output ecr_repository_url`
- `CICD_ROLE_ARN` — from `terraform output cicd_role_arn`
- `APP_POD_ROLE_ARN` — from `terraform output app_pod_role_arn`
- `TF_STATE_BUCKET` — from Step 0's `state_bucket_name` output
- `TF_LOCK_TABLE` — from Step 0's `lock_table_name` output

**Secrets tab**:
- `SONAR_TOKEN` — from your SonarQube/SonarCloud project
- `SONAR_HOST_URL` — e.g. `https://sonarcloud.io` or your self-hosted URL

No AWS access keys are needed anywhere — the pipeline authenticates via
OIDC using `CICD_ROLE_ARN`.

Push to `main`. The pipeline runs, in order:
1. flake8 + pytest
2. SonarQube scan + quality gate (fails build on new vulnerabilities/smells)
3. Docker build → **Trivy scan** (fails build on CRITICAL/HIGH CVEs) → push to ECR
4. Terraform plan → manual approval gate → apply
5. `helm upgrade --install` to EKS (dev values, all account-specific values from repo Variables)

kubectl get pods -n autoforge

kubectl get networkpolicy -n autoforge
kubectl logs -n autoforge deploy/autoforge-app

Get the ALB's auto-generated address (no domain needed):
```powershell
kubectl get ingress autoforge-ingress -n autoforge
```
Look at the `ADDRESS` column — something like
`k8s-autoforge-xxxxxxxx.ap-south-1.elb.amazonaws.com`. Browse to
`http://<that-address>/login` (plain HTTP for now, no ACM cert yet).

The WAF still protects this — test with `' OR 1=1 --` in the username
field and you should get a 403 from WAF before it ever reaches Flask,
even without HTTPS. Once you add a domain later, set
`ingress.acmCertArn` and `ingress.host` via `--set`, `helm upgrade`
again, and you'll get HTTPS with automatic HTTP→HTTPS redirect on the
same ALB.

## Design choices that keep your account ID out of this repo

| Where it used to appear | How it's avoided now |
|---|---|
| Terraform S3 backend bucket name | Partial backend config — bucket/table names passed via `-backend-config` flags at `terraform init` time, not committed |
| Helm `values.yaml` image repo / role ARNs | Left blank (`""`); supplied via `--set` or a gitignored `values-local.yaml` |
| GitHub Actions workflow | Reads `ECR_REPOSITORY`, `CICD_ROLE_ARN`, `APP_POD_ROLE_ARN`, `TF_STATE_BUCKET`, `TF_LOCK_TABLE` from repo-level **Variables**, not hardcoded |
| S3 state bucket uniqueness | A random 8-character suffix (`random_id` resource) instead of the account ID |

## What changed vs. your earlier setup (root-cause fixes)

| Old problem | Fix in this bundle |
|---|---|
| `values-dev.yaml` placeholders silently overwrote secrets on `helm upgrade` | DB creds removed entirely from all values files; ESO owns the Secret object |
| Username mismatch (`autoforge_admin` vs `admin`) | Single source of truth: `var.db_username` in Terraform → Secrets Manager → ESO. Never typed twice |
| Broad/shared IAM roles | Three scoped roles: EKS node, CI/CD (OIDC, resource-scoped), app pod (IRSA, read one secret) |
| No TLS, no WAF | ACM cert (once you have a domain) + WAF v2 (Common, SQLi, KnownBadInputs, rate-limit) attached to the ALB |
| Any pod could talk to any pod | Default-deny NetworkPolicy + explicit allow rules |
| No RBAC | Namespace-scoped Roles for app SA (near-zero), CI/CD deployer, and read-only developers |
| No vulnerability/quality gating | Trivy blocks CRITICAL/HIGH CVEs; SonarQube quality gate blocks vulnerabilities/high-complexity code — both fail the pipeline, not just warn |
